title: "Row-Level Security"
description: "PostgreSQL row-level security helpers for MikroORM entities, request context values, and generated migrations."
---

The `@nest-boot/row-level-security` package helps Nest Boot applications use PostgreSQL Row-Level Security (RLS) with MikroORM. It provides policy metadata for entities, migration helpers that generate SQL from that metadata, and a PostgreSQL driver that applies request-scoped roles and context values inside transaction-local query execution.

## Installation

```bash
pnpm add @nest-boot/row-level-security @nest-boot/request-context @mikro-orm/postgresql @mikro-orm/migrations
```

## Configure MikroORM

Use `RowLevelSecurityDriver` as the MikroORM PostgreSQL driver and `RowLevelSecurityMigrationGenerator` as the migration generator:

```typescript
import {
  RowLevelSecurityDriver,
  RowLevelSecurityMigrationGenerator,
} from "@nest-boot/row-level-security";

export default {
  driver: RowLevelSecurityDriver,
  migrations: {
    generator: RowLevelSecurityMigrationGenerator,
  },
};
```

The driver applies RLS only when a `RequestContext` is active and the current context has row-level security values set through `RowLevelSecurity`. If no row-level security role or context values are present, it delegates to MikroORM unchanged.

## Define Entity Policies

Attach `@Policy()` to MikroORM entities. When `property` and `context` are provided together, the package generates `USING` and `WITH CHECK` predicates that compare the mapped database column with `app.get_context(...)`.

```typescript
import { Entity, PrimaryKey, Property, t } from "@mikro-orm/postgresql";
import {
  Policy,
  PolicyCommand,
  RowLevelSecurityRole,
} from "@nest-boot/row-level-security";

@Policy({
  command: PolicyCommand.ALL,
  property: "workspaceId",
  context: "workspace_id",
  roles: [RowLevelSecurityRole.AUTHENTICATED],
})
@Entity({ tableName: "files" })
export class File {
  @PrimaryKey({ type: t.integer })
  id!: number;

  @Property({ fieldName: "workspace_id", type: t.integer })
  workspaceId!: number;

  @Property({ type: t.string })
  name!: string;
}
```

This policy allows authenticated requests to read, insert, update, and delete rows only when the row's `workspace_id` matches the transaction-local `app.workspace_id` context value.

For custom policy logic, provide explicit SQL predicates:

```typescript
@Policy({
  name: "published_files_read_policy",
  command: PolicyCommand.SELECT,
  using: `"is_public" = true or (select app.get_context('workspace_id', null::integer)) = "workspace_id"`,
  roles: [
    RowLevelSecurityRole.ANONYMOUS,
    RowLevelSecurityRole.AUTHENTICATED,
  ],
})
```

## Generate Migrations

With `RowLevelSecurityMigrationGenerator` configured, MikroORM migration generation includes RLS SQL for policies discovered from entity metadata. Generated migrations extend `RowLevelSecurityMigration`, which adds the shared `app.get_context(...)` helper, database roles, grants, and policy DDL.

```bash
pnpm mikro-orm migration:create
```

Generated policy migrations:

1. Enable row-level security on the target table.
2. Create `anonymous` and the roles declared by policy metadata during `up`.
3. Drop and recreate policies when metadata changes.
4. Grant the required role, schema, table, and sequence privileges during `up`.
5. Revoke the grants emitted by the migration during `down`, without dropping database roles.
6. Rebuild the previous policy state in the down migration.

## Set Request Context

`RowLevelSecurityDriver` applies RLS by running `SET LOCAL ROLE ...` and `set_config('app.<key>', value, true)` inside the active transaction. It reads request-scoped values written through `RowLevelSecurity`.

```typescript
import {
  RowLevelSecurity,
  RowLevelSecurityRole,
} from "@nest-boot/row-level-security";
import { RequestContext } from "@nest-boot/request-context";

import { WorkspaceMember } from "./workspace-member.entity";

const workspaceMember = RequestContext.get(WorkspaceMember);

if (workspaceMember) {
  RowLevelSecurity.setRole(RowLevelSecurityRole.AUTHENTICATED);
  RowLevelSecurity.setContext("workspace_id", workspaceMember.workspace.id);
  RowLevelSecurity.setContext("workspace_member_id", workspaceMember.id);
}
```

Context keys must be `snake_case`, because they are exposed as PostgreSQL settings under the `app.` prefix.

RLS mode is request-scoped behavior. It affects only queries executed while a `RequestContext` is active; outside a `RequestContext`, the driver delegates to MikroORM unchanged.

| Mode | Behavior |
| --- | --- |
| `RowLevelSecurityMode.AUTO` | Default. Applies RLS when a role or at least one context value is set. If RLS is applied and no role was set, the driver uses `anonymous`. With no role and no context values, it delegates unchanged. |
| `RowLevelSecurityMode.ENABLED` | Forces RLS for the current `RequestContext`, even when there are no context values. If no role was set, the driver uses `anonymous`. |
| `RowLevelSecurityMode.DISABLED` | Bypasses RLS for the current `RequestContext`, even when role or context values are set. Use only for trusted infrastructure, admin, or cross-tenant job work. |

`anonymous` is a special database role exported as `RowLevelSecurityRole.ANONYMOUS`. Generated migrations always create it during `up`, because it is the fallback role whenever RLS is enabled but no explicit role was set. Other roles are read from `@Policy({ roles })`, including custom role names, and are also created during `up`. Add `roles: [RowLevelSecurityRole.ANONYMOUS]` to policies that should allow public or unauthenticated access, and call `RowLevelSecurity.setRole(RowLevelSecurityRole.AUTHENTICATED)` for authenticated tenant-scoped work.

```typescript
import {
  RowLevelSecurity,
  RowLevelSecurityMode,
} from "@nest-boot/row-level-security";

RowLevelSecurity.setMode(RowLevelSecurityMode.DISABLED);
```

If a queue job needs MikroORM's request context but should operate across tenants, create the `RequestContext` as usual and do not set `RowLevelSecurity` values. For tenant-scoped work inside a job, set the role and tenant context around that tenant's work:

```typescript
import {
  RowLevelSecurity,
  RowLevelSecurityRole,
} from "@nest-boot/row-level-security";

await RequestContext.run(new RequestContext({ type: "job" }), async () => {
  RowLevelSecurity.setRole(RowLevelSecurityRole.AUTHENTICATED);
  RowLevelSecurity.setContext("workspace_id", workspaceId);

  await processWorkspace(workspaceId);
});
```

## Runtime Behavior

When a query runs outside a transaction, `RowLevelSecurityDriver` opens a short transaction, applies the RLS role and context, executes the SQL, and lets commit or rollback clear transaction-local settings. If MikroORM already has a transaction context, the driver applies the same setup to that transaction before running the query.

Because the setup happens at the SQL connection layer, lazy-loaded relations use the normal request entity manager and receive a fresh RLS transaction when they execute later. You can still call `em.transactional()` directly when several operations must share one RLS context.

## Best Practices

1. **Prefer Generated Policies** — Use `property` + `context` for tenant or workspace isolation so migration SQL stays tied to entity metadata.

2. **Keep Context Small** — Store stable identifiers such as `workspace_id`, `tenant_id`, and `user_id`, not whole objects.

3. **Use Explicit Roles** — Include `roles` on policies so grants and PostgreSQL role behavior stay predictable.

4. **Bypass Deliberately** — For trusted cross-tenant jobs, use `RequestContext` for MikroORM but do not set `RowLevelSecurity`.
