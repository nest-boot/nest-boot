---
title: "Row-Level Security"
description: "面向 MikroORM 实体、请求上下文和迁移生成的 PostgreSQL 行级安全辅助工具。"
---

`@nest-boot/row-level-security` 包帮助 Nest Boot 应用在 MikroORM 中使用 PostgreSQL Row-Level Security（RLS）。它提供实体策略元数据、根据元数据生成 SQL 的迁移辅助工具，以及在事务级查询执行中应用请求作用域角色和上下文值的 PostgreSQL 驱动。

## 安装

```bash
pnpm add @nest-boot/row-level-security @nest-boot/request-context @mikro-orm/postgresql @mikro-orm/migrations
```

## 配置 MikroORM

将 `RowLevelSecurityDriver` 配置为 MikroORM 的 PostgreSQL 驱动，将 `RowLevelSecurityMigrator` 配置为迁移扩展，并将 `RowLevelSecurityMigrationGenerator` 配置为迁移生成器：

```typescript
import {
  RowLevelSecurityDriver,
  RowLevelSecurityMigrationGenerator,
  RowLevelSecurityMigrator,
} from "@nest-boot/row-level-security";

export default {
  driver: RowLevelSecurityDriver,
  extensions: [RowLevelSecurityMigrator],
  migrations: {
    generator: RowLevelSecurityMigrationGenerator,
    snapshot: false,
  },
};
```

策略差异检测目前要求 `migrations.snapshot: false`。开启 MikroORM snapshot 模式时，生成的 policy migration 不会写入 schema snapshot；如果在应用该迁移之前再次运行 `migration:create`，可能会再次生成同一个 policy diff。在支持基于 snapshot 的 RLS policy tracking 前，请保持关闭 snapshot。

驱动只会在 `RequestContext` 处于活动状态，并且当前上下文中通过 `RowLevelSecurity` 设置了行级安全值时应用 RLS。如果没有设置 RLS role 或 context 值，它会按 MikroORM 原始逻辑执行；但如果同一个事务之前已经应用过 RLS 状态，驱动会先清理这些状态。

## 定义实体策略

在 MikroORM 实体上添加 `@Policy()`。同时提供 `property` 和 `context` 时，包会生成 `USING` 和 `WITH CHECK` 谓词，将映射后的数据库列与事务级 `app.<key>` 设置比较。

```typescript
import { Entity, PrimaryKey, Property, t } from "@mikro-orm/postgresql";
import {
  Policy,
  PolicyCommand,
  RowLevelSecurityRole,
} from "@nest-boot/row-level-security";

@Policy({
  command: PolicyCommand.ALL,
  property: "workspaceId",
  context: "workspace_id",
  roles: [RowLevelSecurityRole.AUTHENTICATED],
})
@Entity({ tableName: "files" })
export class File {
  @PrimaryKey({ type: t.integer })
  id!: number;

  @Property({ fieldName: "workspace_id", type: t.integer })
  workspaceId!: number;

  @Property({ type: t.string })
  name!: string;
}
```

这个策略只允许已认证请求读取、插入、更新和删除 `workspace_id` 与事务级 `app.workspace_id` 上下文一致的行。

需要自定义策略逻辑时，可以直接提供 SQL 谓词：

```typescript
@Policy({
  name: "published_files_read_policy",
  command: PolicyCommand.SELECT,
  using: `"is_public" = true or (select nullif(current_setting('app.workspace_id', true), '')::integer) = "workspace_id"`,
  roles: [
    RowLevelSecurityRole.ANONYMOUS,
    RowLevelSecurityRole.AUTHENTICATED,
  ],
})
```

显式的 `using` 和 `withCheck` 可以写成 `true` 或 `"is_public" = true` 这样的裸 SQL 片段；生成迁移时会把它们包装成 PostgreSQL policy 谓词。

## 预置数据库角色

RLS 运行时会执行 `SET LOCAL ROLE ...`，因此数据库中的 RLS roles 必须由数据库管理员提前创建，并授予给应用运行时使用的登录角色。生成的迁移不会创建这些 roles，也不会执行 role membership 授权。

例如，使用内置的 `anonymous` 和 `authenticated` roles 时，数据库管理员可以先执行：

```sql
do $$ begin if not exists (select 1 from pg_roles where rolname = 'anonymous') then create role anonymous nologin; end if; end $$;
do $$ begin if not exists (select 1 from pg_roles where rolname = 'authenticated') then create role authenticated nologin; end if; end $$;
grant anonymous, authenticated to app_user;
```

将 `app_user` 替换为应用连接 PostgreSQL 时使用的登录角色。如果 `@Policy({ roles })` 中声明了自定义 role，也需要用同样方式预先创建，并授予给应用登录角色。

## 生成迁移

配置 `RowLevelSecurityMigrationGenerator` 后，MikroORM 生成迁移时会根据实体元数据中的策略写入 RLS SQL。生成的迁移会继承 `RowLevelSecurityMigration`，用于添加策略 DDL，以及显式 policy role 需要的 table 或 sequence 授权。
`RowLevelSecurityMigrator` 会让 `migration:create` 检测只有策略变化、表结构不变的场景，例如只修改 policy 的 `context`。

```bash
pnpm mikro-orm migration:create
```

生成的策略迁移会：

1. 在目标表上启用 row-level security。
2. 假定 `anonymous` 和 policy metadata 声明的 roles 已经由数据库管理员预先创建。
3. 在元数据变化时删除并重建策略。
4. 在 `up` 中授予需要的 table 和 sequence 权限。
5. 在 `down` 中只撤销不再被旧策略需要的 table、sequence 授权，但不删除数据库 role，也不撤销 role membership。
6. 在 down 迁移中恢复之前的策略状态。

## 设置请求上下文

`RowLevelSecurityDriver` 会在活动事务内执行 `SET LOCAL ROLE ...` 和 `set_config('app.<key>', value, true)` 来应用 RLS。它会读取通过 `RowLevelSecurity` 写入的请求作用域值。

```typescript
import {
  RowLevelSecurity,
  RowLevelSecurityRole,
} from "@nest-boot/row-level-security";
import { RequestContext } from "@nest-boot/request-context";

import { WorkspaceMember } from "./workspace-member.entity";

const workspaceMember = RequestContext.get(WorkspaceMember);

if (workspaceMember) {
  RowLevelSecurity.setRole(RowLevelSecurityRole.AUTHENTICATED);
  RowLevelSecurity.setContext("workspace_id", workspaceMember.workspace.id);
  RowLevelSecurity.setContext("workspace_member_id", workspaceMember.id);
}
```

上下文 key 必须是 `snake_case`，因为它们会以 `app.` 前缀暴露为 PostgreSQL 设置。

RLS mode 是请求作用域行为。它只影响 `RequestContext` 活动期间执行的查询；在 `RequestContext` 外部，驱动会按 MikroORM 原始逻辑执行，除非该查询复用了已经被驱动应用过 RLS 状态的事务。

| Mode                            | 行为                                                                                                                                                                                                    |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `RowLevelSecurityMode.AUTO`     | 默认模式。设置了 role 或至少一个 context 值时应用 RLS。如果应用 RLS 但没有显式设置 role，驱动会使用 `anonymous`。如果没有 role 也没有 context 值，则按原始逻辑执行。                                    |
| `RowLevelSecurityMode.ENABLED`  | 对当前 `RequestContext` 强制应用 RLS，即使没有任何 context 值。如果没有显式设置 role，驱动会使用 `anonymous`。                                                                                          |
| `RowLevelSecurityMode.DISABLED` | 对当前 `RequestContext` 绕过 RLS，即使已经设置了 role 或 context 值。如果当前事务已经应用过 RLS 状态，驱动会先清理这些事务级状态，再执行 disabled 查询。只应在可信的基础设施、管理或跨租户 job 中使用。 |

`anonymous` 是一个特殊数据库角色，并以 `RowLevelSecurityRole.ANONYMOUS` 导出。当 RLS 被启用但没有显式设置 role 时，它就是默认 fallback role。其他 role 会从 `@Policy({ roles })` 读取，包括用户自定义 role 名称。需要允许公开或未登录访问的策略可以添加 `roles: [RowLevelSecurityRole.ANONYMOUS]`；已认证的租户作用域工作则调用 `RowLevelSecurity.setRole(RowLevelSecurityRole.AUTHENTICATED)`。

```typescript
import {
  RowLevelSecurity,
  RowLevelSecurityMode,
} from "@nest-boot/row-level-security";

RowLevelSecurity.setMode(RowLevelSecurityMode.DISABLED);
```

如果 queue job 需要 MikroORM 的请求上下文，但任务本身要跨租户运行，可以照常创建 `RequestContext`，但不要设置 `RowLevelSecurity`。如果 job 内部要按租户执行，则在该租户的执行边界内设置 role 和租户上下文：

```typescript
import {
  RowLevelSecurity,
  RowLevelSecurityRole,
} from "@nest-boot/row-level-security";

await RequestContext.run(new RequestContext({ type: "job" }), async () => {
  RowLevelSecurity.setRole(RowLevelSecurityRole.AUTHENTICATED);
  RowLevelSecurity.setContext("workspace_id", workspaceId);

  await processWorkspace(workspaceId);
});
```

## 运行时行为

查询在事务外运行时，`RowLevelSecurityDriver` 会自动开启一个短事务，应用 RLS 角色和上下文，执行 SQL，并通过提交或回滚清理事务级设置。如果 MikroORM 已经有事务上下文，驱动会在该事务上应用同样的设置后再执行查询。如果复用该事务时禁用了 RLS、清空了请求作用域的 RLS 值，或下一条查询在没有 `RequestContext` 的情况下复用该事务，驱动会在执行查询前清理之前应用的 role 和 context 设置。共享同一事务的查询会由驱动串行化，确保每条查询都使用为它准备的 role 和 context 执行。

因为设置发生在 SQL 连接层，之后 lazy-load relation 仍会使用正常的请求实体管理器，并在实际发起查询时获得新的 RLS 事务。当多个操作必须共享同一个 RLS 上下文时，也可以直接调用 `em.transactional()`。

## 最佳实践

1. **优先使用生成策略** — 对租户或工作区隔离使用 `property` + `context`，让迁移 SQL 与实体元数据保持一致。

2. **保持上下文小而稳定** — 存储 `workspace_id`、`tenant_id`、`user_id` 等标识符，不要存储整个对象。

3. **显式指定角色** — 在策略上包含 `roles`，让授权和 PostgreSQL 角色行为更可预测。

4. **有意绕过** — 可信跨租户后台任务可以继续使用 MikroORM 需要的 `RequestContext`，但不要设置 `RowLevelSecurity`。
