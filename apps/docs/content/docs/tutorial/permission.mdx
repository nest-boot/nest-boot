---
title: "Permission"
description: "CASL-based route permission checks with request-scoped ability caching."
---

The `@nest-boot/permission` module provides a small CASL integration for NestJS applications. It lets you build one permission ability per request, cache it in `RequestContext`, and declare route requirements with the `Can` decorator.

## Installation

```bash
pnpm add @nest-boot/permission @nest-boot/request-context @casl/ability
```

## Module Registration

Register `PermissionModule` with a `buildAbility` callback. The callback receives the current request context with `req` and, when available, `res`. It can return a CASL ability or `null`. Returning `null` causes guarded routes to fail with `ForbiddenException`.

```typescript
import {
  PermissionAbilityBuilder,
  PermissionAction,
  PermissionModule,
} from "@nest-boot/permission";
import { RequestContextModule } from "@nest-boot/request-context";
import { Global, Module } from "@nestjs/common";

import { Post } from "./post.entity";

@Global()
@Module({
  imports: [
    RequestContextModule,
    PermissionModule.forRoot({
      buildAbility: ({ req }) => {
        const user = req.user;

        if (!user) {
          return null;
        }

        const { can, build } = new PermissionAbilityBuilder();

        can(PermissionAction.READ, Post);
        can(PermissionAction.UPDATE, Post, { authorId: user.id });

        return build();
      },
    }),
  ],
})
export class CommonModule {}
```

## Register the Guard

`PermissionGuard` is provided by `PermissionModule`. Register it as a global guard when routes should enforce `Can` metadata.

```typescript
import { PermissionGuard } from "@nest-boot/permission";
import { Module } from "@nestjs/common";
import { APP_GUARD } from "@nestjs/core";

@Module({
  providers: [
    {
      provide: APP_GUARD,
      useExisting: PermissionGuard,
    },
  ],
})
export class AppModule {}
```

## Protect Routes

Use `Can` with a static subject class when the permission check only depends on the resource type.

```typescript
import { Can, PermissionAction } from "@nest-boot/permission";
import { Query, Resolver } from "@nestjs/graphql";

import { Post } from "./post.entity";

@Resolver(() => Post)
export class PostResolver {
  @Can(PermissionAction.READ, Post)
  @Query(() => [Post])
  posts() {
    return this.postService.findAll();
  }
}
```

Use a subject hook when the ability needs the concrete resource instance. The hook receives the decorated resolver or controller method parameters directly. For GraphQL resolvers, `@Args("id") id: string` is passed as the `id` argument below. For HTTP controllers, parameters such as `@Param("id") id: string` and custom parameter decorators are resolved the same way.

```typescript
import { Can, CanSubjectHook, PermissionAction } from "@nest-boot/permission";
import { Injectable } from "@nestjs/common";
import { Args, Mutation, Resolver } from "@nestjs/graphql";

import { Post } from "./post.entity";

@Injectable()
export class PostSubjectHook implements CanSubjectHook<Post, [string]> {
  constructor(private readonly postService: PostService) {}

  run(id: string) {
    return this.postService.findOneOrFail({ id });
  }
}

@Resolver(() => Post)
export class PostResolver {
  constructor(private readonly postService: PostService) {}

  @Can(PermissionAction.UPDATE, Post, PostSubjectHook)
  @Mutation(() => Post)
  updatePost(@Args("id") id: string) {
    return this.postService.update(id);
  }
}
```

## Check Permissions in Services

After `PermissionGuard` has run, the resolved ability is stored in `RequestContext`. Use `can` for a synchronous check in lower-level code.

```typescript
import { can, PermissionAction } from "@nest-boot/permission";
import { ForbiddenException, Injectable } from "@nestjs/common";

import { Post } from "./post.entity";

@Injectable()
export class PostService {
  update(post: Post) {
    if (!can(PermissionAction.UPDATE, post)) {
      throw new ForbiddenException();
    }

    // Update the post.
  }
}
```

`getPermissionAbility()` is also available when direct CASL access is needed. It throws `ForbiddenException` if the ability has not been prepared for the current request.
